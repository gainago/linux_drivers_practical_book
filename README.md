## 2. Kprobes и перехват системных вызовов(./Drivers_from_vm/kprobe)

Один из самых интересных проектов в репозитории это перехват системных вызовов (в частности, `write`) с использованием **Kprobes**.
Интересный он (для меня) потому что реализовать его пришлось без огладки на гайд.

### Контекст
В 10-й главе LKMPG описывается классический метод перехвата syscall: поиск `sys_call_table`, отключение флага защиты от записи `WP` в регистре `cr0` и подмена указателя на функцию. 
> Но в ядре 6.8.0 этот метод не подходит: начиная с версии 5.7, может вызвать Kernel Panic.

### Решение через Kprobes

1. Kprobes находит адрес символа `__x64_sys_write` в памяти ядра и вставляет туда breakpoint.
2. pre и post handler:
    `handler_pre` срабатывает до выполнения системного вызова `write`.
    `handler_post` срабатывает после того, как функция отработает.
3. Внутри хендлеров мы получаем доступ к `struct pt_regs *regs`. Это позволяет читать `regs->ip`, `regs->flags`, а также регистры `di`, `si`, `dx`, в которых лежат аргументы, переданные пользовательской программой в `write` (дескриптор файла, буфер, количество байт).


**Дополнение:** Включает Python-скрипты (`script_add_module_to_initramfs.py` и `script_remove_module_from_initramfs.py`) для автоматизации внедрения `.ko` файлов прямо в `initramfs`. Это подгружает драйверы на самых ранних этапах boot-процесса, еще до монтирования корневой файловой системы, чтобы отлаживать их прямо на начальном этапе загрузки.

### 3. `./Drivers_from_vm/manual`
Эта директория содержит адаптированные под современное ядро 6.8.0 примеры из руководства. Переписывал и немного изменял драйверы из мануала, чтобы лучше понять как они устроены, каждый подпроект демонстрирует работу с внутренним API ядра:

* **`manual_examples`** (Глава 4): Базовые примеры `hello-1` ... `hello-5` и `startstop`. Демонстрируют точку входа/выхода, макросы `__init`/`__exit`, лицензирование и передачу аргументов через `module_param`.

* **`chardev`** (Глава 6): Реализация простейшего символьного устройства. Включает регистрацию номера, работу со структурой `file_operations`.

* **`proc`** (Глава 7): Создание файлов в `/proc`. Показан безопасный обмен данными через `copy_to_user`/`copy_from_user`, а также использование современного API `seq_file` (и макроса `HAVE_PROC_OPS` для совместимости с ядрами 5.6+).

* **`sysfs`** (Глава 8): Взаимодействие с модулем через атрибуты в `sysfs` (`/sys/kernel/...`). Демонстрируется создание кастомных `kobject` и использование макросов `__ATTR` для чтения/записи переменных ядра.

* **`ioctl`** (Глава 9): Взаимодействие через системный вызов `ioctl`. Реализована генерация команд с помощью макросов `_IOW`/`_IOR` и безопасная передача структур данных.

* **`wait`** (Глава 11.1): Механизмы ожидания. Демонстрируется использование очередей ожидания (WaitQ) и перевод процессов в сон (`TASK_INTERRUPTIBLE`) до освобождения ресурса.

* **`completion`** (Глава 11.2): Синхронизация потоков ядра с помощью механизма завершений (completions), когда один поток ждет сигнала от другого.

* **`mutex` и `spinlock`** (Главы 12.1, 12.2): Примитивы синхронизации для избежания race conditions. Мьютексы для операций с возможностью засыпания, спин-блокировки (с `irqsave`) для коротких критических секций в атомарном контексте.

* **`schedule`** (Глава 14): Отложенное выполнение задач с помощью очередей заданий (workqueues) как современной и безопасной альтернативы устаревающим тасклетам.

* **`print_strint_to_tty`** (Глава 13.1): Нестандартный вывод информации. Модуль перехватывает структуру `tty_struct` текущей задачи и выполняет прямую запись строк в терминал через низкоуровневый вызов `tty_operations->write`.

* **`hash`** (Глава 16.1): Использование ядерного крипто-API (`crypto_alloc_shash`) для вычисления хешей (SHA256) и симметричного шифрования (AES) непосредственно в пространстве ядра.

* **`input_device`** (Глава 17): Полноценный драйвер виртуального устройства ввода (`vinput` и `vkbd`). Эмуляция нажатий через `input_report_key` и управление жизненным циклом через `sysfs`.


# 4. Отладка модулей ядра Linux в QEMU/KVM с GDB
> Полная конфигурация моей витуальной машины приведена в debian13.xml.
## 1. Подготовка окружения виртуализации

### 1.1. Установка пакетов QEMU/KVM

```bash
sudo apt install qemu-kvm qemu-utils libvirt-daemon-system libvirt-clients
sudo apt install virtinst virt-manager ovmf
```
## 2. Настройка совместного доступа к файлам (9P VirtIO)

```xml

<filesystem type='mount' accessmode='mapped'>
  <driver type='path'/>
  <source dir='/debian_13'/>
  <target dir='host9p'/>
  <address type='pci' domain='0x0000' bus='0x07' slot='0x00' function='0x0'/>
</filesystem>
```

* source dir - путь к папке на хост-системе
* target dir - метка для монтирования в гостевой системе
* accessmode='mapped' - режим доступа (безопасный)

### 2.2. Монтирование в гостевой системе

```bash 
# Создание точки монтирования
sudo mkdir -p /mnt/shared

# Монтирование shared папки
sudo mount -t 9p -o trans=virtio,version=9p2000.L host9p /mnt/shared

# Автоматическое монтирование при загрузке (/etc/fstab)
host9p /mnt/shared 9p trans=virtio,version=9p2000.L,rw,_netdev 0 0
```

**Назначение** : Эта папка используется для:
* Размещения скомпилированных .ko файлов модулей
* Передачи образа ядра с отладочной информацией (vmlinux)

## 3. Настройка QEMU как GDB сервера

```xml

<qemu:commandline>
  <qemu:arg value='-s'/>
  <qemu:arg value='-S'/>
</qemu:commandline>
```

**Назначение параметров**:
-s - открывает TCP порт 1234 для подключения GDB (эквивалент -gdb tcp::1234)
-S - замораживает CPU при старте ВМ до подключения отладчика

## 4. Подготовка ядра и модулей

* Cкачайте ядро и соберите с отладочной информацией. `CONFIG_DEBUG_INFO=y`, после получения `vmlinux`, поместите его в разделяемую папку. Установите ядро в систему.
Перезагрузите vm и в GRUB выберите ваше ядро, собранное с отладочной информацией.

* Соберите модуль с отладочной информацией. К примеру
`./Drivers_from_vm/manual/sysfs/Makefile`
> Примечание: лучше не использовать в имени модуля символ `-` (тире)

```Makefile
ccflags-y := -g -DDEBUG 
```
И поместите его в разделяемую папку.

## 5. Отладка модулей и ядра

### 5.1 На хосте выполните
```bash
* Запуск GDB с файлом ядра (vmlinux)
gdb /debian_13/vmlinux

* В GDB подключаемся к QEMU
(gdb) target remote :1234
```

### 5.2 На vm

```bash
# Загрузка модуля
sudo insmod /mnt/shared/hello_sysfs.ko

# Проверка загрузки
lsmod | grep hello_sysfs

# В гостевой системе - получение адреса секции .text
cat /sys/module/hello_sysfs/sections/.text
```

И получаете адрес, по которому ядро загрузило ваш модуль, к примеру `0xffffffffc0a01000`

`!!!ВНИМАНИЕ!!!` Если вы помечаете функции init end cleanup соответствующими тегами __init and __exit, то их код располагается в секциях .init and .exit, а не в .text

### 5.3 На хосте

```bash
(gdb) add-symbol-file /debian_13/hello_sysfs.ko -s .text 0xffffffffc0a01000
```
Проверить что всё хорошо, вы можете командами 
``` bash
(gdb) info address "function_in_your_module"
(gdb) list "function_in_your_module"
```
> Так же будьте внимательны, gdb ищет исходники по тому же адресу, по которому они были на момент компиляции модуля
>> Можете прописать замены путей в ~/.gdbinit
``` bash
#~/.gdbinit
set substitute-path /home/gainageorgy/Drivers/manual/sysfs/         /debian_13/modules/source
```

* пример с несколькими секциями

``` bash
(gdb) add-symbol-file /debian_13/hello_sysfs.ko 0xffffffffc0a01000 \
    -s .data 0xffffffffc0a02000 \
    -s .bss 0xffffffffc0a03000 
```

Потом ставити breakpoint и продолжаете выполнение.
При вызове соответствующей функции модуля vm остановится и вы сможете отлаживать драйвер черех gdb, так же возможно перейти в функции ядра и посмотреть их.
